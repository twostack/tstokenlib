## The Problem

TSL1 is a token protocol on BSV where smart contracts enforce token rules entirely on-chain. The PP3 contract (partial witness check) needs to verify that the *current transaction* matches expected constraints — that the right inputs are being spent, that the right outputs are being created, and that the witness chain is intact.

But Bitcoin Script has no opcode that says "give me the current transaction's data." A script can verify a signature against the transaction (`OP_CHECKSIG`), but it cannot directly read the transaction fields (inputs, outputs, amounts, etc.) that it is executing within.

The original sCrypt-compiled PP3 contract was ~100KB of hex — a massive opaque blob generated by the sCrypt compiler. We wanted to replace it with a hand-optimized, dynamically generated script using our own SHA256 implementation. This required re-implementing every component of the contract from scratch in raw Bitcoin Script opcodes, including the `checkPreimageOCS` mechanism that is the subject of this document.

## Background: Sighash Preimage

Before understanding checkPreimageOCS, we need to understand what a sighash preimage is.

When `OP_CHECKSIG` executes, it does not verify a signature against arbitrary data. It computes a *sighash* — a SHA256d hash of a structured serialization of the current transaction — and verifies the signature against that hash. The structured serialization is called the **sighash preimage**, and for BSV (BIP-143 / SIGHASH_FORKID), it has this format:

```
nVersion           (4 bytes)
hashPrevOuts       (32 bytes) — SHA256d of all input outpoints
hashSequence       (32 bytes) — SHA256d of all input sequence numbers
outpoint           (36 bytes) — txid + output index being spent
scriptCode         (variable) — the locking script (subscript)
value              (8 bytes)  — satoshi value of the output being spent
nSequence          (4 bytes)  — sequence number of this input
hashOutputs        (32 bytes) — SHA256d of all outputs
nLockTime          (4 bytes)  — locktime
sighashType        (4 bytes)  — sighash flags
```

This preimage completely and uniquely determines the transaction from the perspective of a given input. If you know the preimage, you know everything about the transaction: which UTXOs it spends, what outputs it creates, what amounts are involved.

## The Key Insight: OP_PUSH_TX

The technique known as **OP_PUSH_TX** (coined by sCrypt / nChain) exploits a circular relationship:

1. `OP_CHECKSIG` computes `z = SHA256d(preimage)` internally and verifies a signature `(r, s)` against it.
2. If we could construct a valid signature *inside the script* from a *claimed* preimage, and then have `OP_CHECKSIG` verify it, the verification would only succeed if our claimed preimage matches the actual transaction.

In other words: we construct a signature from `SHA256d(claimed_preimage)`, then `OP_CHECKSIG` independently computes `SHA256d(actual_preimage)` and verifies. If the signature passes, the two hashes must be equal, which means `claimed_preimage == actual_preimage`.

This turns `OP_CHECKSIG` from a simple signature verifier into a **preimage oracle** — it tells the script whether the data on the stack is an accurate representation of the current transaction.

## How checkPreimageOCS Works

"OCS" stands for **OP_CODESEPARATOR** — we'll see why shortly. The algorithm uses a **known private key and known nonce** to construct an ECDSA signature entirely within Bitcoin Script.

### The Constants

Six values are embedded in the script as constants:

| Constant | Description |
|----------|-------------|
| `privKey` | A known ECDSA private key (secp256k1 scalar) |
| `invK` | The modular inverse of a known nonce `k`, i.e. `k^(-1) mod N` |
| `r` | The x-coordinate of `k * G` on secp256k1 (the `r` component of the signature) |
| `pubKey` | The public key corresponding to `privKey` (a compressed EC point) |
| `N` | The order of the secp256k1 curve |
| `rBigEndian` | The `r` value in big-endian format for DER encoding |

These are not secret. Anyone can see them in the script. The security does not depend on the private key being secret — it depends on the mathematical relationship between the signature and the sighash being unforgeable without knowing the actual transaction data.

### The Algorithm (High Level)

```
1. preImage is on the stack (pushed by the scriptSig)
2. z = HASH256(preImage)                    // claimed sighash
3. z_int = fromBEUnsigned(z)               // convert to a script number
4. s = invK * (z_int + r * privKey) mod N  // ECDSA signature formula
5. Normalize s to low-S form               // BIP-62 / BIP-146
6. DER-encode (r, s) with sighash byte     // build the signature blob
7. OP_CODESEPARATOR                        // change the subscript boundary
8. OP_CHECKSIG with pubKey                 // verify!
```

If the preImage is correct (matches the actual transaction), then `z` equals the actual sighash that `OP_CHECKSIG` will compute, the constructed signature will be valid, and `OP_CHECKSIG` returns TRUE.

If the preImage is wrong, `z` will be different from the actual sighash, the constructed signature will not verify, and `OP_CHECKSIG` returns FALSE (or the script aborts).

## The ECDSA Math In Detail

### Standard ECDSA Signing

In standard ECDSA signing (outside of script), given:
- Private key `d`
- Message hash `z`
- Random nonce `k`

The signature `(r, s)` is computed as:

```
R = k * G                    (elliptic curve point multiplication)
r = R.x mod N                (x-coordinate of R)
s = k^(-1) * (z + r * d) mod N
```

### In-Script Signing

We pre-compute `k`, `r`, `k^(-1)`, and `d` offline. These become constants in the script. The only variable is `z` (the hash of the claimed preimage), which is computed on-stack.

The script computes:

```
s_raw = invK * (z + r * privKey)
```

Using Bitcoin Script opcodes:

```
OP_HASH256              // z = hash256(preImage)
reverseBytes32          // z in little-endian
CAT 0x00                // append sign byte (ensure positive)
OP_BIN2NUM              // z as a script number

PUSH rLE                // push r (little-endian script number)
PUSH privKeyLE          // push privKey (little-endian script number)
OP_MUL                  // r * privKey
OP_ADD                  // z + r * privKey
PUSH invKLE             // push k^(-1)
OP_MUL                  // invK * (z + r * privKey) = s_raw
```

### Modular Reduction

The raw `s_raw` may exceed `N`, so we normalize it:

```
s_mod = s_raw % N
if s_mod < 0:
    s_mod = s_mod + N    // ensure positive
```

In script:

```
PUSH N
OP_2DUP                 // [s_raw, N, s_raw, N]
OP_MOD                  // [s_raw, N, s_mod]
OP_DUP
OP_0
OP_LESSTHAN             // is s_mod negative?
OP_IF
    OP_OVER             // copy N
    OP_ADD              // s_mod + N
OP_ENDIF
OP_NIP OP_NIP           // clean up → [s_normalized]
```

Why might `s_mod` be negative? Bitcoin Script's `OP_MOD` can return negative values when the dividend is negative. Although our hash is always positive (we appended `0x00`), the intermediate products of large multiplications can produce negative script numbers depending on internal representation. The conditional `+N` handles this edge case.

### Low-S Normalization

BIP-62 / BIP-146 require the `s` value to be in the lower half of the curve order. If `s > N/2`, we replace it with `N - s`:

```
PUSH N
OP_2DUP                 // [s, N, s, N]
OP_2 OP_DIV             // [s, N, s, N/2]
OP_GREATERTHAN          // [s, N, s > N/2]
OP_IF
    OP_SWAP OP_SUB      // N - s
OP_ELSE
    OP_DROP             // drop N, keep s
OP_ENDIF
```

## The OP_CODESEPARATOR Trick

Here is the subtlest part of the entire mechanism.

### The Problem

When `OP_CHECKSIG` computes the sighash, it includes the **scriptCode** field in the preimage. The scriptCode is derived from the locking script, specifically the portion after the most recently executed `OP_CODESEPARATOR`.

If our script is 62KB and the preimage must include the entire 62KB as the scriptCode, then the preimage itself would be enormous, and computing `SHA256d(preimage)` would require processing all that data.

But the scriptSig's preimage was constructed *before* execution, using the full locking script as the subscript. The dartsv library's `createSighashPreImage()` strips the subscript up to the first `OP_CODESEPARATOR`:

```dart
// From dartsv Sighash.createSighashPreImage():
for (var chunk in subscriptCopy.chunks) {
    if (chunk.opcodenum == OpCodes.OP_CODESEPARATOR) {
        firstOpCodeSep = chunkIndex + 1;
        break;
    }
    chunkIndex++;
}
var newSubScript = SVScript.fromChunks(
    subscriptCopy.chunks.sublist(firstOpCodeSep));
```

### The Solution

We place `OP_CODESEPARATOR` immediately before `OP_CHECKSIG`:

```
<pubKey>
OP_CODESEPARATOR
OP_CHECKSIG
```

When `OP_CHECKSIG` executes, it uses the scriptCode from after the most recently executed `OP_CODESEPARATOR`. This means the scriptCode is just:

```
OP_CHECKSIG [OP_ELSE ... OP_ENDIF]
```

A tiny fragment instead of the full 62KB script.

The preimage that was pushed in the scriptSig was constructed with this same stripped subscript (thanks to `createSighashPreImage` stripping up to the first `OP_CODESEPARATOR`). So the `z = HASH256(preimage)` computed in-script matches the `z_actual` computed by `OP_CHECKSIG`. The signature verifies.

### Why "OCS"

The name `checkPreimageOCS` means "check preimage using OP_CODESEPARATOR." Without the CODESEPARATOR trick, the preimage would need to include the full script as the scriptCode field, making the technique impractical for large scripts (and creating a circular dependency — the script would need to contain its own hash).

## DER Encoding In Script

ECDSA signatures in Bitcoin must be DER-encoded. Our script must construct this encoding on-stack from the computed `s` value and the constant `r` value.

### DER Signature Format

```
0x30 <total_len>
  0x02 <r_len> <r_bytes>
  0x02 <s_len> <s_bytes>
0x41                          // sighash type (SIGHASH_ALL | SIGHASH_FORKID)
```

### Construction in Script

The `r` component is constant, so we pre-encode it as a fixed byte string: `0x02 0x20 <32-byte r in big-endian>`.

The `s` component is variable (depends on the hash). The script:

1. **Gets the minimal byte length of `s`**: `OP_DUP OP_SIZE OP_NIP` — script numbers are minimally encoded, so SIZE gives us the right length.

2. **Converts to 32-byte big-endian**: `OP_NUM2BIN(32)` gives 32-byte LE, then `reverseBytes32` flips to BE.

3. **Trims to minimal length**: `OP_SPLIT` at `32 - slen` and drop the leading zeros.

4. **Builds the s TLV**: Concatenates `0x02 || slen_byte || s_BE`.

5. **Prepends the r component**: Concatenates the pre-encoded r TLV.

6. **Wraps in the SEQUENCE header**: Computes the inner length, prepends `0x30 || inner_len_byte`.

7. **Appends sighash type**: Concatenates `0x41`.

The result is a valid DER-encoded signature ready for `OP_CHECKSIG`.

## Number Representation Challenges

Bitcoin Script numbers are little-endian with a sign bit in the MSB of the last byte. This creates several challenges when performing 256-bit elliptic curve arithmetic:

### Unsigned 256-bit Values

SHA256 produces 32-byte big-endian hashes. To use these as script numbers for multiplication:

```
reverseBytes32(hash)     // BE → LE (32 bytes)
CAT 0x00                 // append zero byte to ensure positive sign
OP_BIN2NUM               // convert to minimal script number
```

The appended `0x00` byte is critical. Without it, if the hash's most significant bit is set (50% of the time), `BIN2NUM` would interpret it as a negative number, producing an incorrect ECDSA signature.

### The OCS Constants

All six constants (`privKey`, `invK`, `r`, `N`, `pubKey`, `rBigEndian`) are stored as little-endian byte arrays with a trailing `0x00` sign byte where needed:

```dart
// r as LE script number with sign byte:
static final _rLE = hex.decode(
    '6c266b30e6a1319c66dc401e5bd6b432ba49688eecd118297041da8074ce081000');
//                                                                        ^^
//                                                          sign byte ensures positive
```

### Script Number Overflow

BSV (post-Genesis) has no limit on script number size, so the intermediate products of 256-bit multiplications (which can be up to 512 bits) are handled natively. This is essential — without big number support, the ECDSA math would be impossible in script.

## Our Implementation

The complete PP3 contract is generated by `WitnessCheckScriptGen.generate()` in Dart. It produces a ~62KB Bitcoin Script that implements two functions:

### Unlock Function (selector = OP_0)

The unlock path verifies a witness transaction:

1. **Partial SHA256**: Completes a SHA256 hash from a midstate + remaining blocks. Uses the hand-optimized `Sha256ScriptGen.emitOneBlock()` (~30KB per block, up to 2 blocks inlined). This computes the witness transaction hash.

2. **Outpoint Verification**: Extracts `myOutpoint` from the sighash preimage (bytes 68-104) and verifies that the witness transaction was spending from the expected PP2 output of this same token transaction.

3. **HashPrevOuts Verification**: Reconstructs the expected `hashPrevOuts` from the funding TxID (from scriptSig), the computed witness TxID, and the extracted outpoint. Double-SHA256 hashes them and compares against the hashPrevOuts field in the preimage.

4. **checkPreimageOCS**: The mechanism described in this document. Constructs an ECDSA signature from `HASH256(preimage)` and verifies it with `OP_CHECKSIG`.

### Burn Function (selector = OP_1)

A simple P2PKH check — verifies that the spender's public key hashes to `ownerPKH`, then checks the signature.

### Stack Management

The script uses the altstack to shuttle values across the SHA256 computation:

```
Main stack entry:  [preImage, partialHash, witnessPreImage, fundingTxId, ownerPKH]
                                                                          ↓ DROP
Altstack saves:    [fundingTxId, witnessPartialOutpoint, preImage]

  ← SHA256 computation (uses altstack internally but leaves it clean) →

Altstack after:    [fundingTxId, witnessPartialOutpoint, preImage]
                   Recovered one-by-one for verification steps.
```

`emitOneBlock()` has zero net altstack effect — it pushes W_blob, K_blob, and midstate_copy during compression but pops all three by the end.

## Why This Was Necessary

The sCrypt-compiled PP3 template was ~100KB of opaque hex. By dynamically generating the script, we achieved:

1. **38% size reduction**: ~62KB vs ~100KB, saving on-chain fees.

2. **Transparency**: Every opcode is accounted for in readable Dart code with stack annotations. The sCrypt compilation was a black box.

3. **Parameterization**: The `pp2OutputIndex` parameter allows the same generator to produce PP3 scripts for different transaction layouts (NFT at index 2, FT at index 5, etc.).

4. **Independence from sCrypt toolchain**: No compiler dependency for generating or understanding the contract.

The NFT variant retains the old compiled template because the PP1 inductive proof contract has the expected PP3 script format embedded in its verification logic. The FT variant uses the new dynamic generator since the FT contracts (PP1_FT, PP2-FT) are being built to work with it.

## Bugs Encountered

### dartsv `addData([0x02])` Assertion Failure

When building the DER `s` component, we needed to push the literal byte `0x02` (the ASN.1 INTEGER tag). `addData(Uint8List.fromList([0x02]))` caused an `IllegalArgumentException`.

**Root cause**: dartsv's `ScriptBuilder.insertData()` has special handling for single-byte values 1-16: it converts them to `OP_N` opcodes (e.g., value 2 becomes `OP_2`, opcode 0x52). However, it still creates a `ScriptChunk` with a non-null buffer alongside the OP_N opcode. When `SVScript._convertChunksToByteArray()` encounters an opcode chunk (opcodenum > 78) with a non-null buffer, it fails an assertion.

**Fix**: Use `b.smallNum(2)` instead of `b.addData([0x02])`. `smallNum` correctly creates a chunk with null buffer. Functionally equivalent — `OP_2` pushes the number 2, which serializes as byte `0x02` when concatenated with `OP_CAT`.

### Burn Path Stack Order

The burn path initially had `[pubKey, sig]` on the stack before `OP_CHECKSIG`. But `OP_CHECKSIG` expects `[sig, pubKey]` — the public key on top.

**Fix**: Added `OP_SWAP` before `OP_CHECKSIG` in the burn path.

### PP1 Inductive Proof Incompatibility

After replacing the NFT PP3 template with the dynamic generator, the PP1 spending tests for token transfers failed. The PP1 contract (a separate sCrypt-compiled script) embeds knowledge of the expected PP3 script format as part of its inductive proof — it verifies the parent token transaction's outputs, which includes PP3.

Since our dynamically generated PP3 produces different bytes than the sCrypt-compiled version, PP1's verification fails. The NFT `PartialWitnessLockBuilder` was reverted to the old hex template. The FT builder uses the new generator since the FT contracts are built from scratch.
